The current TOC implementation reaches the correct href, but the first click lands at a mid-section CFI. The second click on the same TOC item lands at the section start.

Evidence:

resultCfi: epubcfi(/6/32!/4/4/2/2/1:0)

Same pattern for split_012 and split_013.

Conclusion:
The href is resolved correctly, but display(index) / display(href) is not a reliable "chapter start" command on first call. It appears to preserve or reuse an intra-section page offset until the section is already active.

Do not fix this by requiring a double click.
Do not add more restore timers, stabilization windows, or page-1 save guards.

Patch TOC navigation only:

For TOC hrefs without a fragment:
1. Resolve the spine section inside WebView:
   const section = book.spine.get(targetHref)

2. Load the section if needed:
   await section.load(book.load.bind(book))

3. Compute an explicit start CFI for the section:
   const doc = section.document
   const startElement =
     doc.body.querySelector('h1,h2,h3,h4,p,section,div') ||
     doc.body.firstElementChild ||
     doc.body
   const startCfi = section.cfiFromElement(startElement)

4. Navigate using:
   await rendition.display(startCfi)

For TOC hrefs with a #fragment:

Do not use rendition.display(section.index) for TOC no-fragment items anymore.
